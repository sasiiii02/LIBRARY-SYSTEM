## 1. Introduction

This report documents the development of a Library Management System, a full-stack web application that allows users to perform CRUD operations on book records. The project was completed as part of the internship assignment for Expernetic.

## 2. Technology Choices

### Backend: C# .NET 8
- **Why .NET?** Robust, type-safe, excellent for RESTful APIs with built-in dependency injection
- **Entity Framework Core** - ORM that simplifies database operations and provides LINQ support
- **SQLite** - Lightweight, file-based database perfect for this scope, requires no separate server
- **Swagger/OpenAPI** - Automatic API documentation for easy testing and integration

### Frontend: React + TypeScript
- **React** - Component-based architecture for reusability and maintainability
- **TypeScript** - Type safety reduces runtime errors and improves developer experience
- **Tailwind CSS** - Utility-first styling for rapid UI development without context switching
- **Axios** - Promise-based HTTP client with interceptors and better error handling
- **Lucide React** - Beautiful, consistent icons that are tree-shakeable

## 3. Implementation Details

### 3.1 Backend Architecture

The backend follows clean architecture principles with separation of concerns:

- **Models** (`Book.cs`) - Database entities representing the core domain
- **DTOs** (`CreateBookDto.cs`, `BookResponseDto.cs`) - Data transfer objects for API contracts, ensuring internal fields aren't exposed
- **Controllers** (`BooksController.cs`) - Handle HTTP requests/responses with proper status codes
- **Data Context** (`AppDbContext.cs`) - Database connection and operations with Entity Framework
- **Program.cs** - Application configuration, CORS setup, and service registration

**Key Features:**
- RESTful endpoints following conventions (`/api/books`, `/api/books/{id}`)
- Async/await for non-blocking database operations
- Proper HTTP status codes (200 OK, 201 Created, 404 Not Found, 400 Bad Request)
- Input validation on required fields (Title is mandatory)
- Automatic database creation with seed data (The Hobbit, 1984)
- CORS configured to allow React frontend on port 5173

### 3.2 Frontend Architecture

The frontend is organized into a clean, scalable structure:

- **Components** - Reusable UI pieces (`BookForm`, `BookList`, `ConfirmModal`)
- **API Layer** (`booksApi.ts`) - Centralized API calls with Axios
- **Context** (`ToastContext.tsx`) - Global state management for notifications
- **Types** (`Book.ts`) - TypeScript interfaces ensuring type safety

**Key Features:**
- **Loading States**: Spinners during form submission, skeletons during initial fetch
- **Error Handling**: Try-catch blocks with user-friendly toast messages
- **Form Validation**: Real-time validation with visual feedback and error messages
- **Search/Filter**: Client-side filtering without extra API calls (case-insensitive)
- **Toast Notifications**: Global success/error messages that auto-dismiss after 3 seconds
- **Responsive Design**: Mobile-first approach with Tailwind breakpoints
- **Delete Confirmation**: Modal dialog to prevent accidental deletions

## 4. Challenges Faced and Solutions

### Challenge 1: CORS Configuration
**Problem:** React app couldn't access the API due to CORS policy blocking requests from different origins.
**Solution:** Configured CORS in .NET with `AllowAnyOrigin()`, `AllowAnyMethod()`, and `AllowAnyHeader()` during development. In production, this would be restricted to specific domains.

### Challenge 2: Type Safety Between Frontend and Backend
**Problem:** Ensuring frontend types match backend responses to prevent runtime errors.
**Solution:** Created TypeScript interfaces (`Book.ts`) that mirror backend DTOs. Used `Omit<Book, "id">` for create operations where ID is generated by the database.

### Challenge 3: State Management During Edit Mode
**Problem:** Managing multiple books in edit mode simultaneously was complex with separate state variables.
**Solution:** Used `editingId` to track which book is being edited, with separate `editData` state for form values. This allows only one book to be edited at a time, simplifying state management.

### Challenge 4: Toast Notification System
**Problem:** Needed global notifications without prop drilling through multiple components.
**Solution:** Implemented React Context API (`ToastContext`) with a custom `useToast` hook. This provides toast functionality anywhere in the app while keeping the implementation encapsulated.

### Challenge 5: Database File Location
**Problem:** SQLite database file was being created in different locations depending on how the app was run.
**Solution:** Explicitly set the database file path in `Program.cs` to ensure consistent location in the project folder.

## 5. Key Insights Gained

1. **Full-Stack Integration**: Understanding how data flows from React form → API call → .NET controller → database and back requires careful coordination of types and error handling.

2. **TypeScript Benefits**: Catching type errors at compile time rather than runtime saved hours of debugging. The `Omit` utility type was particularly useful for create operations.

3. **Context API vs Props**: For global concerns like notifications, Context API is cleaner than prop drilling. For component-specific state, local useState is sufficient.

4. **Async/Await Pattern**: Proper async/await usage with try/catch/finally ensures loading states are always reset, even when errors occur.

5. **User Experience Matters**: Small touches like loading spinners, confirmation modals, and toast notifications significantly improve perceived performance and user satisfaction.

6. **DTO Pattern**: Separating database entities from API responses prevents over-exposing internal data and gives flexibility to shape responses as needed.

7. **Entity Framework**: The ORM simplifies database operations significantly, but understanding the difference between `EnsureCreated()` and migrations is important for production scenarios.
